# Releasing

Source of truth: `.github/workflows/release.yml`, `.github/workflows/ci.yml`, `cliff.toml`.

## CI

Every push to `main` and every pull request runs the CI workflow (`.github/workflows/ci.yml`). It builds drift, runs `drift lint`, then cross-compiles for all 4 targets (aarch64-macos, x86_64-macos, x86_64-linux, aarch64-linux). Build depends on lint passing.

## Releasing a version

Releases use conventional commits and git-cliff for changelog generation.

### Commit format

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
feat: add python symbol resolution
fix: handle empty frontmatter gracefully
docs: update design doc with new data model
refactor: extract scanner into separate module
```

Types `chore`, `style`, and `ci` are excluded from changelogs. Merge commits are also skipped.

### Creating a release

1. Check what the next version should be: `git-cliff --bumped-version`
2. Tag and push:
   ```
   jj tag create vX.Y.Z
   jj git push --tags
   ```
3. The tag push triggers `.github/workflows/release.yml`, which:
   - Generates release notes with git-cliff (grouped by Features, Bug Fixes, Documentation, Refactor)
   - Cross-compiles for all 4 targets with Zig 0.15.2
   - Creates a GitHub release with the generated notes and all tarballs attached

### CHANGELOG.md

`CHANGELOG.md` is generated by git-cliff from the commit history. It will be created on the first release that uses conventional commits. To regenerate locally:

```
git-cliff -o CHANGELOG.md
```
